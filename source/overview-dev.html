FHIR Overview - Developers

FHIR (Fast Health Interoperability Resources) is designed to enable the information exchange that supports the provision of healthcare in a wide variety of processes. The specification builds on and adapts standard industry RESTful practices to enable the provision of integrated healthcare across a wide range of teams.

Framework

FHIR is based on "Resources" which are the common building block for all exchanges. Each resource has the following common features:

  • A URL that identifies it
  • A set of defined common data elements
  • A human-readable XHTML summary
  • An extensibility framework to support variation in healthcare

Resources are represented as either XML or JSON. There are <%resourcecount%> different resource types defined in the FHIR specification.

Example Patient

This is an example of how a patient is represented as a FHIR object in JSON. XML encoding is also defined in the specification.

Interactions

For manipulation of resources, FHIR provides a REST API with a rich but simple set of interactions:

  • Create = POST https://example.com/path/{resourceType}
  • Read = GET https://example.com/path/{resourceType}/{id}
  • Update = PUT https://example.com/path/{resourceType}/{id}
  • Delete = DELETE https://example.com/path/{resourceType}/{id}
  • Search = GET https://example.com/path/{resourceType}?search parameters...
  • History = GET https://example.com/path/{resourceType}/{id}/_history
  • Transaction = POST https://example.com/path/
  • Operation = GET https://example.com/path/{resourceType}/{id}/${opname}

The FHIR specification describes other kinds of exchanges beyond the simple API outlined above, including exchange of groups of resources as Documents, Messages, and by using other kinds of services.

Managing Variability

One feature of the healthcare industry is that there is a wide variation between different jurisdictions and sections of the industry, and no central authority to impose common business practices. Because of this, the FHIR specification defines a common extension framework, and defines several resources for managing this variability:

  • Conformance (OPTIONS https://example.com/path/)

    Get a conformance statement that describes the capabilities of the system

  • Profile (GET /Profile)

    Descriptions of the how the system uses resources for a specific purposes.

  • Value Sets (GET /ValueSet)

    Formal description of how Terminologies and Codes systems are used with resources

  • Operation definitions (GET /OperationDefinition)

    Introspection of additional RPC-like operations the system offers in addition to the basic RESTful API operations

Create Response

A response contains an HTTP code 201 to indicate that the Resource has been created successfully. A location header indicates where the resource can be fetched in subsequent requests. The server may choose to return the created resource (which may differ from what was submitted), but is not required to do so. 

HTTP/1.1 201 Created
Content-Length: 161
Content-Type: application/json+fhir; charset=UTF-8
Date: Mon, 18 Aug 2014 01:43:30 GMT
ETag: 1
Location: http://example.com/Patient/347

{
  "resourceType": "OperationOutcome",
  "text": {
    "status": "generated",
    "div": "<div xmlns=\"http://www.w3.org/1999/xhtml\">The operation was succesful</div>"
  }
}

Get Request

Reading a resource is done by sending HTTP GET requests to the desired Resource end point, as in this example. Note the use of _format=xml - this is another method for clients to indicate the desired response format, in addition to using the accept header, and is useful for clients that don't have access to the HTTP Headers. 

GET /Patient/347?_format=xml HTTP/1.1
Host: example.com
Accept: application/xml+fhir
Cache-Control: no-cache

Get Response

The response to a GET contains the Resource. The content location can be used for version aware updates. 

HTTP/1.1 200 OK
Content-Length: 729
Content-Location: http://example.com/Patient/347/_history/1

Last-Modified: Sun, 17 Aug 2014 15:43:30 GMT

Search Request

In addition to getting single Resources it is possible to fetch sets of Resources by searching the resource end point with a set of criteria describing the set of resources that should be retrieved, and their order. In addition, the search operation is also able to return additional related resources as well.

  • https://example.com/{resourceType}?criteria
  • https://example.com/MedicationPrescription?patient=347 - return all the prescriptions for the patient created above
