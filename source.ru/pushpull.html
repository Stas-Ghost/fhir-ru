Управление доставкой информации по технологиям Push и Pull

Основное внимание FHIR направлено на обмен медицинскими данными между системами. Всякий раз, когда у одной системы имеется информация, которая требуется или необходима другой, один вопрос, который автоматически возникает, это система-источник доставит данные без запроса (технология push) или система назначения извлечет эти данные по запросу (технология pull).

Технология Push Как только становится доступной новая информация или всякий раз, когда она сочтет это целесообразным, система-источник посылает эту информацию получателю. Она ожидает, что система-получатель будет поддерживать и индексировать эту информацию по получении. Система-источник должна доверить системе-получателю управление надлежащим образом доступом/безопасностью.
Технология Pull Система-источник поддерживает и индексирует данные. Когда системе-получателю требуется или необходима эта информация, она должна извлечь ее из системы-источника. Либо источник, либо получатель (либо оба) управляет доступом/безопасностью.

Обратите внимание, что имеется возможность объединять эти режимы добавлением репозитория интерфейсов (interface repository) в этот смешанный состав. Например, чтобы связать источник, работающий по технологии push с получателем, работающим по технологии pull, просто настройте источник доставлять информацию в репозиторий без запроса (push), и получатель сможет извлекать данные из него по необходимости (pull). Если источник работает по технологии pull, репозиторий интерфейсов может регулярно посылать запрос на новые данные и, обнаружив их, стянуть из репозитория (pull) и/или доставить (push) получателю.

Какая комбинация технологий push, pull и репозиториев интерфейсов является подходящей для данной ситуации, зависит от массы факторов: архитектуры развертывания, топологии сети и бизнес-конфигурации, политики безопасности и проч. Однако она часто ограничивается возможностями различных систем и стандартов, используемых при обмене.

FHIR-спецификация поддерживает и разрешает использование обеих технологий push и pull:

  • REST: REST API может использоваться в обоих способах действия - это обсуждается более подробно ниже.
  • Messages: FHIR устанавливает события обмена сообщениями для push и pull.
  • Services: Сервисы могут устанавливаться для поддержки любой из моделей.
  • Documents: Этот вопрос не применим, т. к. документы являются комплектами, которыми обмениваются с помощью либо REST, либо сообщений, либо сервисов.

Технологии Push и Pull в RESTful-интерфейсе

  • Push: Источником является клиент, и когда доступны новые данные, он использует create, update и transaction для доставки данных получателю (push), которым является сервер.
  • Pull: Источником является сервер, а получателем - клиент. Когда клиенту необходимо использоваться данные, он использует комбинацию search и read таким образом, чтобы пользователи могли искать данные, которые им нужны.
  • Push/Pull: Источником является сервер, а получателем - клиент. Получатель регулярно запрашивает у сервера его историю изменений и копирует изменения в свою собственную копию базы данных.
