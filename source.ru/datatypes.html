Типы данных

В спецификации FHIR описаны типы данных, которые используются в элементах ресурсов. Типы данных делятся на две категории: простые / примитивные типы, являющиеся отдельными элементами, и сложные типы, являющиеся повторно используемыми кластерами элементов. На этой странице приведены описания этих типов данных. Также см. страницы Примеры, Подробное описание и Мэппинги.

Примечания:

  • Для всех типов их XML и JSON представления совпадают за исключением различий в экранировании символов в XML и JSON
  • Приведенные регулярные выражения пропускают более широкий набор значений, чем фактически допустим (например високосные года), поэтому требуется дополнительная валидация
  • Точность десятичных значений существенна:
    • например 0.010 рассматривается как отличное от 0.01
    • Реализации ДОЛЖНЫ обрабатывать десятичные значения таким образом, чтобы сохранять и учитывать точность представленного значения
    • Реализации, выполняющие это ограничение, являются GMP-реализациями или Java BigDecimal, который реализует произвольную точность, либо комбинацией значения с плавающей точкой (64 bit) с полем для указания точности
    • Обратите внимание, что нет абсолютного лимита для размера значения, хотя большие и/или высокоточные значения чрезвычайно редки в медицине. Единственный элемент, где могут встретиться десятичные значения высокой точности - это координаты в Location

XML и JSON представления

Все элементы, использующие эти примитивные типы, имеют некоторую комбинацию значения, как описано выше, внутренней сущности (например xml:id) и расширения. Значение представляется в XML как атрибут "value": 

  <count value="2"/>

и как значение свойства в JSON: 

  "count" : 2

Дополнительную информацию включая представление внутренней сущности и расширений см. на страницах описания форматов XML и JSON. Когда значение отсутствует, оно не обозначается в экземпляре; XML-атрибут value или JSON-свойство в этом случае не указываются вообще.

В соответствии с XML-схемой пробелы в начале и в конце атрибута value игнорируются для типов boolean, integer, decimal, base64Binary, instant, uri, date, dateTime, oid и uri. Обратите внимание, из этого следует, что библиотеки XML, следующие схеме, дают разные значения атрибутов по сравнению с библиотеками, не соответствующими схеме, во время чтения экземпляров XML. По этой причине атрибут value для этих типов НЕ ДОЛЖЕН иметь пробелы в начале и в конце. Строковые значения могут иметь пробелы в начале и в конце, если они являются частью содержимого их значения. В JSON пробелы в строковых значениях всегда являются значимыми. Примитивные типы за исключением string НЕ ДОЛЖНЫ содержать пробелы в начале и в конце.

Простые ограничения

Кроме базовых примитивных типов определено несколько дополнительных примитивных типов в виде ограничений над одним из других базовых типов.

codestringУказывает, что значение взято из набора регулируемых строк, определенных в другом месте (см. Использование кодов для дальнейшего обсуждения). Техническое ограничение: код должен быть строкой, содержащей хотя бы один символ и не имеющей пробела в начале и в конце, внутри которой есть только одиночные пробелы
regex: [^\s]+([\s]+[^\s]+)*

Некоторые дополнительные замечания по типу данных id:

  • Идентификаторы чувствительны к регистру. UUID-ы ДОЛЖНЫ передаваться с использованием строчных букв
  • Тип ID включает в себя идентификаторы в соответствии с ISO 18232, но также включает и другие форматы идентификаторов, но не чувствителен к регистру, как ISO 18232.
  • В типичном FHIR URL, наподобие http://example.com/fhir/Patient/1234, последняя часть "1234" (выделена цветом) имеет тип данных ID
  • Полный UUID - это URI, а не id. UUID-ы в URI также ДОЛЖНЫ быть представлены в нижнем регистре (urn:uuid:59bf0ef4-e89c-4628-9b51-12ae3fdbe22b)

Сложные типы

Эти типы представлены XML-элементами, имеющими элементы-потомки с именем определенных элементов этого типа. (These types are represented as XML Elements with child elements with the name of the defined elements of the type.) Имя элемента определяется там, где этот тип используется. Любой XML-элемент может иметь атрибут id. В JSON этот тип данных представлен объектом со свойствами, называющимися так же, как и эти XML-элементы. JSON-представление почти ничем не отличается, поэтому только первый из примеров содержит дополнительно JSON-представление.

Сложные типы данных могут быть "профилированы". Structure Definition типа "constraint" задает набор правил о том, какие элементы ДОЛЖНЫ иметь значения и какие значения являются допустимыми.

Вложение (Attachment)

См. также Примеры, Подробное описание и Мэппинги.

Тип данных, содержащий вложение или ссылку на вложение - дополнительные данные в другом формате. Наиболее частое использование этого типа - включение изображений или отчетов в некотором формате документов, например PDF. Однако он может использоваться для любых данных MIME-типа.

[%dt Attachment%]

Элемент contentType всегда ДОЛЖЕН иметь значение. Он может содержать информацию о кодировке (charset) и MIME-расширении по мере необходимости. Если в элементе contentType не указана кодировка, тогда правильный курс действий не определен, хотя некоторые медиа-типы могут указывать кодировку по умолчанию и/или правильная кодировка может определяться проверкой содержимого.

Само содержимое вложения (Attachment) может быть передано непосредственно с помощью элемента data, либо через указание ссылки в элементе URL. Если приведено и то, и то, ссылка ДОЛЖНА вести на то же самое содержимое, что и в элементе data. Эта ссылка не может быть повторно использована для указания на какие-то другие данные (например относиться только к определенной версии). Ссылка в элементе URL ДОЛЖНА указывать на место, возвращающее сами данные; некоторые URIs, например cid:, отвечают этому требованию. Если в URL указана относительная ссылка, она интерпретируется таким же образом, как и ссылка на ресурс.

Элемент hash позволяет приложениям проверить, что содержимое, возвращаемое по ссылке в URL, не изменилось.

В большинстве случаев использования вложения (Attachment), количество элементов >1. Допустимое использование повторов позволяет передавать одно и то же содержание на разных языках и MIME-типах. Указания по интерпретации повторяющихся элементов ДОЛЖНЫ быть приведены в определении повторяющегося элемента ресурса или расширения, ссылающегося на этот тип (or extension that references this type). Элемент language описывает язык вложения с помощью кодов, определенных в BCP (Best Current Practice) 47.

Ограничения

[%dt.constraints Attachment%]

Если ни data, ни URL не заполнены, значение следует понимать как утверждение о том, что для такой комбинации language и contentType нет доступных данных.

Контекст использования часто может диктовать правила о том, какой тип вложения (и, следовательно, какой MIME-тип) может быть использован.

Вложение (Attachment) используется в следующих местах: [%dtusage Attachment%]

Coding

См. также Примеры, Подробное описание и Мэппинги.

Coding - это представление определенного концепта с помощью символа из определенной "кодовой системы" - см. Использование кодов в ресурсах для получения более подробной информации.

[%dt Coding%]

Значение Coding определяется кодом. Система (system) указывает на источник определения кода наряду с необязательной ссылкой на версию. Элемент display содержит человекочитаемый текст, определенный системой - он не добавляет другого значения. Ссылка на Value Set дает информацию о контексте выбора этого кода.

Система system - это URI, ссылающийся на кодовую систему, определяющую этот код code. Значением URI может быть:

  • URI из перечня названий систем
  • OID (urn:oid:) либо UUID (urn:uuid:). OIDs и UUIDs могут быть взяты в реестре HL7 OID
  • URL, ссылающийся прямо на определение системы, который может быть ссылкой на кодовую систему, определенную как часть ресурса ValueSet (например значение в ValueSet.define.system)
  • или любой другой URI, уникально идентифицирующий определение кодовой системы

Правильное значение для использования в system для данной кодовой системы может быть определено путем проработки списка ниже в следующем порядке:

  • раздел Перечень названий систем
  • Реестр HL7 OID
  • документация, относящаяся к кодовой системе
  • консультирование с владельцем кодовой системы
  • задать вопрос в списке рассылки справочника HL7

Также может быть указана версия кодовой системы. Если значение кодов в кодовой системе остается постоянным в разных версиях, то это не обязательно. Версию НЕОБХОДИМО менять, когда система больше не поддерживает непротиворечивые определения в разных версиях. Если есть ссылка на набор значений (value set), и этот набор значений определяет кодовую систему, и в этом определении указана версия, то эта версия ДОЛЖНА быть такой же, как и в наборе значений (value set). Обратите внимание, что следующим системам указывать версию НЕОБХОДИМО всегда:

  • Национальные выпуски SNOMED CT (согласованность определений различается в разных странах, и некоторые страны могут принимать свои собственные правила на этот счет)
  • Различные версии ICD (примечание: основные релизы помечены как совершенно разные кодовые системы, но есть различия в версиях)

В общем случае любая классификация (например кодовая система), которая включает концепты с относительными определениями, например "not otherwise coded", потребует указания версии. См. обсуждение версий кодовых систем в ресурсе Value Set для дальнейшего обсуждения версионирования.

Если имеется, code ДОЛЖЕН быть синтаксически правильным символом, как это определено системой system. В некоторых кодовых системах, например SNOMED CT, этот символ может быть выражением, составленным из других предопределенных символов (например post-coordination). Обратите внимание, что коды чувствительны к регистру, если в кодовой системе не указано иное. display - это текстовое представление кода, определенное системой system и используемое для отображения значения кода приложением, которое не знает этой системы system.

Если в кодовой системе имеется несколько возможных строк для отображения, то одна из них ДОЛЖНА быть использована в display. Если одна из них помечена как предпочтительная, то именно она ДОЛЖНА быть использована. Если кодовая система не предлагает текстовое представление (например SNOMED CT Expressions), тогда display не может быть заполнен, и значение кода не будет доступно системам, которые не понимают кодовое выражение.

В некоторых случая система system может не быть известна - известен только код. В этом случае никакая полезная обработка кода не может быть выполнена, если только система не может быть надежно выведена из контекста. Этой практики следует избегать, где это возможно, так как обмен информации в более широком контексте весьма вероятно возникнет в конце концов, и коды не могут использоваться в отсутствие знания о системе.

Если система присутствует, и нет кода, это следует понимать как отсутствие подходящего кода в системе, где должен быть этот код (in which to represent the code).

Если два codings имеют одинаковые систему system, версию version и код code, значит они имеют одинаковое значение. Если информация о версии отсутствует, или элементы система system, версия version или код code различаются, тогда как их коды связаны, может быть определено обращением к определениям этих систем(ы), и любые мэппинги доступны.

Элемент coding может быть помечен как "первичный" ("primary"), если пользователь выбрал определенное кодовое значение в интерфейсе пользователя (например пользователь выбирает пункт в списке). Первичное значение является предпочтительным выбором для выполнения переводом, например.

Ссылка на value set может приводиться в помощь пользователю или системе, обрабатывающей код, понять контекст, в котором этот код возник. В некоторых обстоятельствах возможных набор кодов, из которого этот код был выбран, влияет на то, каким образом этот код надо интерпретировать. Ссылка на набор значений не является допустимым замещением для логического URI в системе. Системный URI НЕ ДОЛЖЕН содержать ссылку на набор значений. (Если этот набор значений определяется свои собственные коды, тогда корректное значения для системы будет ValueSet.define.system, и набор значений содержит прямую ссылку на ресурс value set.

Ограничения

Если valueSet указан, системный URI ДОЛЖЕН быть также приведен.

Контекст использования (как определено в ресурсе или применимом (подходящем) профиле) обычно устанавливает правила о том, какие коды и системы разрешены или требуются в определенном контексте с помощью привязки элемента к набору значений.

Coding используется в следующих местах: [%dtusage Coding%]

Примечание разработчика: в данной спецификации определено два типа представления кодированных значений:

  • Coding: простая непосредственная ссылка на код, определенный кодовой системой
  • CodeableConcept: текстовое описание и/или список Codings (т.е. список ссылок на коды, определенные кодовой системой)

Тип данных Coding соответствует простому случаю выбора одиночного кода из списка. Однако этот тип редко используется в спецификациях FHIR; большой опыт обмена кодированными значениями в HL7 показал, что в общем случае системы должны иметь возможность обмениваться несколькими переводами кодов и/или оригинальным текстом.

Тип данных Coding используется, когда есть уверенность в том, что значение должно быть выбрано непосредственно из доступных кодов, и список возможных кодов согласован со всеми участниками. Обычно это не тот случай в контексте FHIR - общей совместимости - таким образом Coding используется главным образом в расширениях, которые обычно предназначены для определения хорошо контролируемых условий использования.

CodeableConcept

См. также Примеры, Подробное описание и Мэппинги.

CodeableConcept представляет собой значение, которое обычно сопровождается ссылкой на один или несколько справочников или онтологий, но может также определяться предоставлением текста. Это распространенный шаблон представления медицинских данных.

[%dt CodeableConcept%]

Каждый coding является представлением концепта как описано выше. Концепт может быть закодирован несколько раз в различных кодовых системах (или даже несколько раз в одной и той же кодовой системе, где допустимо несколько форм, например SNOMED CT). Различные кодинги могут иметь слегка различающуюся степень разбиения (структурирования, детализации) из-за различий в определениях лежащих в основе кодов. Порядок кодингов внутри CodeableConcept не имеет значения. Типичное использование CodeableConcept - отправка локального кода, которым был закодирован концепт, и также один или несколько переводов в публично определенные кодовые системы, например LOINC or SNOMED CT. Отправка локальных кодов - полезное и важное в целях отладки и ведения контроля за целостностью.

Независимо от того, какие элементы coding присутствуют, текст text - это представление концепта, как введено или выбрано пользователем, и который наиболее точно передает заложенное значение пользователя или концепта. Очень часто текст text совпадает со значением display одного из кодингов. Один из кодингов может быть помечен как первичный - это код или концепт, который пользователь в действительности непосредственно выбрал. Когда ни один из элементов coding не отмечен как первичный, тогда текст (если присутствует) является предпочтительным источником значения.

Quantity (Количество)

См. также Примеры, Подробное описание и Мэппинги.

Измеренная величина (или величина, которая потенциально может быть измерена).

[%dt Quantity%]

Элемент value содержит числовое значение величины, включая подразумеваемую точность (implicit precision). Если компаратор не указан, то значением является указанное значение (т.е. '='). Элемент comparator не может игнорироваться.

Элемент units содержит отображаемую на экране единицу измерения того, что было измерено. Единицы измерений также могут быть закодированы некоторым формальным образом с помощью code и system (см. Coding для получения дополнительной информации о том, как использовать элемент system).

Если единицы измерений можно закодировать с помощью системы UCUM (Unified Code for Units of Measure) и код присутствует, тогда это ДОЛЖЕН быть код UCUM. Если в code указана единица измерения из UCUM, то можно использовать каноническое значение для сравнения величин. Обратите внимание, что элемент units будет часто содержать текст, являющийся фактически допустимой единицей измерения UCUM, но нельзя предполагать, что элемент units действительно содержит допустимую единицу измерения UCUM.

Ограничения

[%dt.constraints Quantity%]

Контекст использования часто может указывать, что это за величина и, следовательно, какие единицы измерений могут применяться. Кроме того, контекст использования может требовать code из определенной system. Контекст использования может также ограничивать значения value или range.

Некоторые вариации типа Quantity

Они используются в качестве типов в модели содержимого ресурсов, но по сути это просто Quantity с некоторыми ограничениями:

[%dt.restrictions Quantity%]

Range (Диапазон)

См. также Примеры, Подробное описание и Мэппинги.

Множество упорядоченных значений величины (Quantity), заданное указанием верхнего и нижнего пределов.

Range обозначает множество возможных значений; обычно применяется одно значение из диапазона (например, "выдать пациенту от 2 до 4 таблеток"). Диапазоны обычно используются в указаниях (instructions).

[%dt Range%]

Элементы units и code/system в элементах low и high ДОЛЖНЫ совпадать. Если элемент low или high отсутствует, это означает, что нижняя или верхняя граница не известна и, следовательно, ни тот, ни другой не является полным диапазоном (neither is the complete range).

Элементы low и high не могут иметь флаг comparator. Обратите внимание, что тип Range не должен использоваться для измерений, выходящих за пределы диапазона: в этом случае следует использовать тип quantity с элементом comparator.

Значения low и high входят в диапазон и предполагается, что они имеют сколь угодно высокую точность. К примеру, диапазон 1.5 - 2.5 включает значения 1.50 и 2.50, но не 1.49 или 2.51.

Ограничения

[%dt.constraints Range%]

Range используется в следующих местах: [%dtusage Range%]

Ratio (Соотношение)

См. также Примеры, Подробное описание и Мэппинги.

Отношение между двумя значениями количества, выраженное в виде числителя и знаменателя.

[%dt Ratio%]

Общие делители в числителе и знаменателе не сокращаются автоматически. Тип данных Ratio используется для титров (например "1:128") и других величин, полученных лабораториями, которые действительно представляют собой соотношения. Ratio - это не просто "структурированные числа" - измерения артериального давления (к примеру, "120/60") не являются соотношениями. Также соотношения используются, когда не следует сокращать числитель и знаменатель. Наиболее распространенным примером этого является соотношение, которое представляет собой стоимость единицы, а числителем является валюта (например 50$/10).

Правильное соотношение имеет и числитель, и знаменатель; однако это не является обязательным, чтобы можно было использовать недопустимые соотношения с указанием дополнительной информации в расширении.

Ограничения

[%dt.constraints Ratio%]

Контекст использования может потребовать определенные типы Quantity для числителя или знаменателя.

Ratio используется в следующих местах: [%dtusage Ratio%]

Period (Период)

См. также Примеры, Подробное описание и Мэппинги.

Период времени, определенный датой/временем начала и окончания.

Период устанавливает диапазон времени. В контексте использования будет указываться, применять ли полный диапазон (например "пациент находился в стационаре больницы данный диапазон времени") или одно значение из указанного периода (например "выдать пациенту между 14 и 16 часами 24-Jun 2013").

[%dt Period%]

Если элемент start отсутствует, значит начало периода не известно. Если элемент end отсутствует, значит период продолжается.

Значение end включает в себя любую подходящую дату/время. К примеру, период с 2011-05-23 по 2011-05-27 включает любое время с начала дня 23 мая до конца дня 27 мая.

Period используется в следующих местах: [%dtusage Period%]

SampledData (Дискретные данные)

См. также Примеры, Подробное описание и Мэппинги.

Данные серии измерений с помощью изделий, имеющие верхний и нижний пределы. В данных может содержаться более одного измерения.

SampledData представляет собой удобный способ управления данными, полученными на изделиях, которые измеряют определенное физическое состояние с высокой частотой. Типичное использование - вывод данных с электрокардиографа.

[%dt SampledData%]

Данные представляют собой множество десятичных значений, разделенных одиночным пробелом (символ Юникода u20). В дополнение к десятичным значениям используются значения "E" (ошибка), "L" (ниже предела обнаружения) и "U" (выше предела обнаружения). Если имеется более одного измерения, различные измерения чередуются - все точки данных для конкретного времени представлены вместе. Значением коэффициента factor по умолчанию является 1.

SampledData используется в следующих местах: [%dtusage SampledData%]

Identifier (Идентификатор)

См. также Примеры, Подробное описание и Мэппинги.

Числовая или буквенно-числовая строка, ассоциированная с отдельным объектом или сущностью внутри данной системы. Обычно идентификаторы используются, чтобы связать содержимое ресурсов и внешнее содержимое, доступное в других фреймворках или протоколах. Идентификаторы ассоциируются с объектами и могут быть изменены или изъяты из обращения вследствие развития систем, из-за ошибок или с течением времени.

[%dt Identifier%]

Система system, на которую ссылается URI, устанавливает, каким образом определяется идентификатор (т.е. каким образом обеспечивается уникальность значения). Это может быть некоторое приложение или признанный стандарт/спецификация, определяющие доступное множество идентификаторов или способ обеспечения их уникальности. Значение value ДОЛЖНО быть уникальным в пределах указанной системы и иметь соответствующее значение, где бы оно ни появлялось. И system, и value всегда чувствительны к регистру.

FHIR сразу задает несколько полезных URIs. OIDs (urn:oid:) и UUIDs (urn:uuid:) могут быть зарегистрированы в Реестре HL7 OID и должны быть, если их содержимое совместно используется или передается между общественными организациями. Если идентификатор сам по себе является глобально уникальным (например OID, UUID или URI без локального окончания), тогда значением системы system SHALL be "urn:ietf:rfc:3986", ДОЛЖНО быть "urn:ietf:rfc:3986", а в value должен быть указан сам URI.

В некоторых случаях система может быть не известна - известно только значение (например простое устройство, сканирующее штрих-коды), или система понимается неявно (простой обмен в ограниченном контексте, зачастую обусловленный считывателями штрих-кодов). В этом случае невозможно выполнить полезное сопоставление значения, если только система не может быть с уверенностью выведена из контекста. Подобную практику следует по возможности избегать, так как со временем растет вероятность совместного использования информации в более широком контексте, а значения без системы по своей природе ограничены в использовании.

Элемент assigner используется для указания, какой реестр/государство/учреждение/пр. назначили идентификатор.

Identifier используется в следующих местах: [%dtusage Identifier%]

HumanName (Имя человека)

См. также Примеры, Подробное описание и Мэппинги.

Имя человека с указанием текстового представления, частей имени и информации об использовании.

Имя можно изменить или отречься от него. У людей могут быть различные имена в разных обстоятельствах. Имена могут быть разбиты на части разного типа, имеющие переменное значение в зависимости от контекста, хотя деление на части не всегда существенно. В личных именах различные части могут быть пронизаны некоторым неявным смыслом; различные культуры придают различную важность частями имени, и степень, с которой система ДОЛЖНА заботиться о частях имени, по всему миру широко варьируется.

[%dt HumanName%]

Для "given names" вместо полного имени можно использовать инициалы, если другой информации не было записано. Системы, функционирующие одновременно для разных культур должны, как правило, полагаться на текстовую форму для представления, и использовать составные части для функциональности индекса/поиска. По этим причинам приложения ДОЛЖНЫ заполнять элемент "text" для устойчивости в будущем.

Приложения, обновляющие имя, ДОЛЖНЫ убедиться, что либо текст соответствует частям имени, либо из них присутствует только что-то одно. Системы, не поддерживающие столько составляющих частей имени, сколько приходит в экземпляре, могут по желанию соединить некоторые части, разделяя их пробелами.

[%dt.constraints Timing%]

Signature

См. также Примеры, Подробное описание и Мэппинги.

Signature содержит электронную цифровую подпись и вспомогательный контекст в доступной FHIR форме. Доказательство невозможности отказа от авторства указывается через XML-Signature BLOB (прим. пер. - binary large object; большой двоичный объект, "блоб").

[%dt Signature%]

Signature.blob - это XML-Signature в кодировке base64. XML-Signature - это Detached Signature, где подписанное содержимое отделено от самой подписи.

Signature должна соответствовать XAdES-X-L для поддержки подписей в долгосрочной перспективе. Спецификация XAdES-X-L добавляет штамп времени подписания, включение сертификата подписи и заявление об аннулировании.

Там, где подписанным содержимым является FHIR-ресурс(-ы), подпись ставится на XML-форму ресурса(-ов).

Содержимое должно быть представлено в канонизированной форме с помощью "Canonical XML 1.1. with Comments".

Подпись должна использовать алгоритм хэширования SHA256. XML-подпись показывает используемый алгоритм. Если предполагается использование других алгоритмов хэширования, приемлемость будет определяться политикой валидации подписи.

Подпись должна включать в себя элемент "“CommitmentTypeIndication"для указания назначения (назначений) подписи. Назначение может быть заверяемым действием, или ролью, связанной с цифровой подписью. Значение нужно брать из ASTM E1762-95(2013). Signature.type содержит те же значения, что и committmentTypeIndication.

Есть три уровня проверки подписи:

  1. проверка того, что сам блок Digital Signature имеет целостность, верификацией подписи через XML-подпись,
  2. подтверждение, что подписавшая сторона подлинна, не была отозвана, и соответствует назначению подписи,
  3. подтверждение, что интересуемое подписанное содержимое не изменялось, с помощью алгоритма хеширования.

Отклонения от этих принципов должны быть выражены в политике сайта, и будет перечислены в блоке XML-Signature. Например, некоторые среды могут выбрать другой профиль XAdES, алгоритм хэширования, идентификатор политики или справочник назначений подписи.

Ограничения

Элементы открытого типа

У некоторых элементов вместо типа стоит символ "*". В таких случаях типом элемента может быть один из следующих:

  • integer
  • decimal
  • dateTime
  • date
  • instant
  • time
  • string
  • uri
  • boolean
  • code - если определение расширения содержит привязку к подходящему набору кодов
  • base64Binary
  • Coding
  • CodeableConcept
  • Attachment
  • Identifier
  • Quantity
  • Range
  • Period
  • Ratio
  • HumanName
  • Address
  • ContactPoint
  • Timing
  • Signature
  • Reference - - ссылка на другой ресурс

Имена таких элементов заканчиваются на "[x]", эта часть заменяется на имя типа данных с большой буквы.

Открытые ссылки используются в следующих местах: [%dtusage *%]
