Логические модули

Каждый ресурс может принадлежать одному или нескольким логическим модулям. Модуль - это логическая группировка ресурсов, имеющих общее свойство. Модули выполняют две основные роли:

  • Функционируют в качестве механизма доступа для быстрого поиска набора связанных ресурсов
  • Обеспечивают определительную основу для быстрого применения контроля доступа к ресурсам

Определения модулей описывают, каким образом определенные экземпляры модулей именуются и идентифицируются, и каким образом системы понимают, какие ресурсы входят в такой модуль. Следующие модули были определены в рамках данной спецификации:

<%compartmentlist%>

Compartments are defined and added the list above when implementer communities identify them as common access points for data. As described below, compartments have both syntactical and logical consequences, and both these aspects of their functionality are evaluated when deciding whether to define compartments.

Using Compartments

Как пример применения модулей, используйте следующий URL для извлечения списка проблем пациента: 

  GET [base]/Patient/[id]/Condition

Additional search parameters can be defined, such as this hypothetical search for acute conditions: 

  GET [base]/Patient/[id]/Condition?code:in=http://hspc.org/ValueSet/acute-concerns

Note that as searches, these are syntactic variations on these two search URLs respectively: 

  GET [base]/Condition?patient=[id]
  GET [base]/Condition?patient=[id]&code:in=http://hspc.org/ValueSet/acute-concerns

However, there is a key difference in functionality between compartment based searches and direct searches with parameters. Consider this search: 

  GET [base]/Patient/[id]/Communication

Because the definition of the patient compartment for Commnunication says that aCommunication resource is in the patient compartment if the subject, sender, or recipient is the patient, the compartment search is actually the same as the union of these 3 searches: 

  GET [base]/Communication?subject=[id]
  GET [base]/Communication?sender=[id]
  GET [base]/Communication?recipient=[id]

There is no way to do this as a single search, except by using the _filter: 

  GET [base]/Communication?_filter=subject re [id] or sender re [id] or recipient re [id]

Further details of searching by compartment are described under the search operation. As a search related operation, the assignment of resources to compartments is only based on the current version of any of the resources involved. Note that contained patient resources cannot create a patient compartment of their own. -->

Модули могут применяться явным образом, как тут, а также могут использоваться неявно. К примеру, если FHIR-сервер предоставляет попациентный просмотр записи, авторизованный пользователь, ассоциируемый с использованием FHIR RESTful API, может быть ограничен в доступе к записям из экземпляра(-ов) модуля, логически связанных с их идентификатором. Вне зависимости от того, используются ли модули явно или неявно, серверам потребуется принять меры, чтобы сделать некоторые ресурсы без прямой ссылки на пациента, доступными для клиента (медикаменты, вещества и т. д.).

Обратите внимание, что ресурсы могут пересекаться между модулями или связывать их. Примерами этого будет Diagnostic Report определяет субъекта, но Observation, на который он ссылается, определяет другого субъекта, или ресурс List ссылается на пункты, которые определяют разных субъектов. Такие поперечные связи могут возникать по многим допустимым причинам, включая:

  • Случаи, когда предметы записи взаимосвязаны - трансплантаты, перинатологии, семейная терапия и т. д.
  • Управление рабочим процессом, где списки действий связывают несколько пациентов и/или специалистов

Учитывая широкое разнообразие вариантов применения и контекстов использования FHIR, модули не определяют, как обрабатывать поперечные связи. Системы могут отклонить ресурсы, удалить их из обоих модулей, поместить их в оба модуля или действовать ещё каким-то другим образом.

Остается на усмотрение сервера, включать ли ресурсы в модуль, если ссылка на этот ресурс, который устанавливает модуль, находится в расширении.

Some resources are not in any compartment, e.g. Medication, Substance, Location. These resources are not directly to a patient or authored record, and are sometimes called 'master files'. Servers will need to make arrangements to make these resources available to the clients that are limited to particular compartments. For example, a Medication resource describes a medication itself and does not link to a patient; however, a resource such as MedicationAdministration connects the Medication (details of what was administered) to the patient (for whom was it administered), and so is required to interpret the administration.
