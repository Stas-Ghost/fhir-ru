Определения ресурса

Данная спецификация определяет ряд различных типов ресурсов, которые могут быть использованы для обмена и/или хранения данных с целью решения широкого диапазона проблем, связанных со здравоохранением, как медицинских, так и административных. Кроме того спецификация описывает несколько различных способов обмена ресурсами.

Ресурс - это сущность, которая:

  • имеет известное обозначение (url), по которому к ней можно обращаться
  • отождествляет себя с одним из типов ресурса, определенных в этой спецификации
  • содержит набор элементов структурированных данных, которые описаны в определении типа ресурса
  • содержит человекочитаемое представление содержимого ресурса в формате XHTML
  • имеет установленную версию, которая меняется, если содержимое ресурса изменилось

Ресурсы имеют разнообразные представления. Ресурс является допустимым, если он подходит под описанные выше критерии и представлен либо в XML, либо в JSON формате в соответствии с правилами, описанными в данной спецификации. Другие представления допустимы, но не описываются этой спецификацией.

Определения

Базовые определения для этой страницы:

Ресурс Экземпляр данных, который хранится или которым обмениваются
Определение ресурса Устанавливает, какие элементы данных являются частью ресурса
Профиль Вводит дополнительные ограничения на элементы данных в определенном контексте использования. Имеется специальный тип ресурса - Profile Resource, который используется для представления профиля

Содержимое ресурса

Все типы ресурсов имеют следующие необязательные или обязательные элементы и свойства:

  • Основной набор определенных элементов данных, характерных для этого типа
  • Расширения - вспомогательные элементы данных, добавленные в реализациях
  • Человекочитаемое описание содержимого ресурса
  • Входящие в состав ресурсы - вспомогательные ресурсы, являющиеся частью идентификационного и транзакционного пространства ресурса
  • Метаданные - важная информация о ресурсе, не являющаяся частью модели содержимого ресурса
  • Теги - ярлыки, прикрепляемые к ресурсам, которые можно использовать для определения дополнительных рабочих характеристик, таких как безопасность, последовательность выполняемых действий и т. п.

Каждый ресурс содержит общий набор элементов (документацию по этому формату см. на странице Resource Definition Format): 

<[Name] xmlns="http://hl7.org/fhir">  
 <extension><!-- 0..*  Extension   See Extensions  --></extension>
 <modifierExtension><!-- 0..*  Extension   See Extensions  --></modifierExtension>
 <language value="[code]"/><!-- 0..1 Human language of the content (BCP-47) -->
 <text><!-- 0..1 Narrative Text summary of resource content, for human interpretation --></text>
 <contained><!-- 0..*  Resource   Contained Resources  --></contained>
 <!-- Defined Data Elements for Resource -->
</[Name]>

Эти элементы всегда ДОЛЖНЫ идти в этом порядке. Базовые элементы, являющиеся общими для всех ресурсов, идут первыми для поддержки совместимости определений для схемы и полученного из UML кода.

Необязательный элемент language устанавливает основной язык ресурса, применяются коды, определенные в BCP 47. Обратите внимание, что не всё содержимое ресурса должно быть на этом языке. Если язык указан, то он также должен быть указан и для текста описательной части (Narrative Text). У языка нет значения по умолчанию, однако оно может вытекать из контекста использования.

Элемент language обеспечивает поддержку индексирования и доступности (например преобразование "текст - речь" использует тег языка). HTML-тег language в описательной части используется во время ее обработки. Тег ресурса language используется для указания языка альтернативных представлений данных ресурса.

Метаданные ресурса

Свойства метаданных являются ключевыми аспектами ресурса и его поведения. По соображениям реализации они представлены вне ресурса:

Элемент метаданныхТипИспользование
Logical Id id Простая строка, обеспечивающая логическую идентичность ресурса. Уникальна в пространстве всех ресурсов того же типа на том же сервере. Постоянна на протяжении всей жизни ресурса на сервере.
Version IdidИзменяется при каждом изменении содержимого ресурса. На него можно ссылаться в ссылке на ресурс. Может использоваться для гарантии, что обновления основаны на последней версии ресурса.
Версия может быть уникальной глобально либо в области действия Logical Id. Идентификаторы версии, как правило, являются либо последовательно увеличивающимися идентификаторами, ограниченными logical id, либо UUID, хотя ни один из этих вариантов не является обязательным.
Last Modified DateinstantИзменяется при каждом изменении содержимого ресурса. Может использоваться системой или человеком для суждения о сроке действия содержимого ресурса.

Logical и Version идентификаторы чувствительны к регистру. Идентификаторы всегда непрозрачны, и системам нет необходимости и не следует пытаться определить их внутреннюю структуру. Идентификатор ДОЛЖЕН всегда быть представлен одинаковым образом в ссылках на ресурсы и в URLs. Идентификаторы могут быть до 36 символов длиной и содержать любую комбинацию строчных букв в коде ASCII (Американский стандартный код обмена информацией), цифр, знаков "-" и ".".

Примечание: эти элементы данных представлены вне ресурса, потому что:

  • Они почти всегда явно заданы в оболочке данных, в частности, они содержатся в заголовках HTTP и atom.
  • Они будут изменяться в процессе управления, перемещения и копирования ресурсов, даже когда само содержимое не меняется (хотя изменение logical id ресурса может потребовать изменения содержимого других ресурсов, которые ссылаются на него)

Полное наименование ресурса представляет собой абсолютный URL, построенный из адреса сервера, на котором он находится, типа ресурса и Logical Id, например http://test.fhir.org/rest/Patient/123 (где 123 - это Logical Id). Обратите внимание, что реализации НЕ ДОЛЖНЫ полагаться на то, что наименование ресурса всегда разрешается сервером в буквальном смысле - он может быть временно не доступен, либо не доступен из-за политики безопасности (например брандмауэры), либо, в некоторых случаях, он может не существовать в действительности (например использование ресурса вне среды RESTful). Ресурсы ссылаются друг на друга по своим наименованиям. Такие ссылки могут быть абсолютными и относительными (см. Ссылки на ресурсы для дальнейшего обсуждения). Копирование и перемещение ресурсов с одного сервера на другой означает, что ресурсы приобретают новое наименование. Для получения дополнительной информации см. Управление наименованиями ресурсов

И Version Id, и Last Modified Date меняются при каждом изменении ресурса. Version Id больше полезен при администрировании вопросов параллелизма и ссылок, зависящих от версии из-за неотъемлемой неопределенности и пределах точности, связанных с датами/временем. Last Modified Date используется человеком для установления логической последовательности информации в ресурсе.

Технические подробности того, как представлять эти элементы метаданных во время обмена в любой среде, где используются ресурсы, будет необходимо решить голосованием. Для получения дополнительной информации см. Использование ресурсов с помощью сервисов.

Межверсионная совместимость

Следующие правила будут применяться после того, как спецификация станет полностью нормативным стандартом. Эти правила гарантируют, что реализации смогут обрабатывать содержимое ресурсов безопасно во время обмена данными между приложениями, использующими различные версии FHIR. Однако в период пробного использования спецификации HL7 может вносить изменения помимо описанных здесь ограничений в ответ на обнаруженные в спецификации проблемы. Приложения могут при желании использовать теги ресурсов, чтобы помочь в этом в период пробного использования.

У содержимого ресурса нет явного признака версии. Став нормативным документом, последующие версии данной спецификации могут внести новые элементы и/или содержимое в любое место содержимого ресурса, однако путь и значение существующих элементов данных не изменится. Любой набор значений или перечень кодов может быть переработан с целью включения дополнительных кодов.

Каждая привязка к набору значений или кодовой системе указывает, будет ли перечень значений расти автоматически с определением новых кодов, может ли перечень быть расширен в будущих версиях спецификации или же перечень не может быть изменен в будущих версиях вообще.

Слой обеспечения соответствия (Обеспечение соответствия и Профиль) имеет обязательные свойства, описывающие версию спецификации FHIR, и их можно использовать для определения, какую версию FHIR применяет реализация.

В типичном сценарии может потребоваться использование разных версий одновременно, поэтому приложениям СЛЕДУЕТ игнорировать элементы, которые они не распознают, если только это не modifierExtensions. Тем не менее, в контексте здравоохранения многие поставщики приложений не склонны рассматривать этот подход из-за опасений по поводу клинического риска или технических ограничений в их программном обеспечении (например обработка на основе схемы). Приложения не обязаны игнорировать неизвестные элементы, однако ОБЯЗАНЫ объявить, будут ли они это делать, в своем заявлении о соответствии.

Дополнительное обсуждение по межверсионным вопросам можно найти здесь: http://wiki.hl7.org/index.php?title=FHIR_interversion_compatibility.
