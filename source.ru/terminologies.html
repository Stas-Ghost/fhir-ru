Использование кодов в ресурсах

Многие ресурсы в FHIR имеют кодированное значение: некоторая фиксированная строка (последовательность символов), определенная где-то в другом месте, которая соответствует некоторому "концепту". Эта последовательность символов и ее значение могут быть определены в одном из следующих мест:

  • Как одно из набора фиксированных значений, приведенных в этой спецификации
  • В информационных документах Интернета RFC (например MIME-тип, язык)
  • В спецификации HL7 (кодовая система v3 или таблица v2)
  • Во внешних справочниках или онтологиях, таких как LOINC или SNOMED CT
  • В локально сохраненном словаре, таблице поиска или перечне в приложении (дополнительную информацию по локально определенным наборам значений см. в "Profiling FHIR"

Все эти способы определения кодов в совокупности называются "кодовыми системами". Этот список далеко не полон; есть много способов определения кодовых систем, и они отличаются друг от друга по сложности и размеру.

В описании стандарта кодированные значения всегда указываются парой "система − код", где система − это URL на кодовую систему, содержащую определение кода. Note that system values are always case sensitive. Different code systems make their own rules as to whether the codes they define are case sensitive or not. Note that all the codes defined by FHIR itself are case sensitive and SHALL be used in the provided case (usually, but not always, lowercase).

Подход FHIR к использованию кодированных значений основывается на базовой структуре, введенной в разделе 5 документа Основополагающие принципы HL7 v3, включая разделение между кодовыми системами и наборами значений.

Если в ресурсах содержатся коды, то используется один из 4 типов данных:

codeЭкземпляр объекта этого типа представляет собой только код. Информация о системе подразумевается: она содержится в определении самого элемента и не указывается в экземпляре.
CodingТип данных, состоящий из двух элементов: код и система. Последний идентифицирует источник определения значения кода.
CodeableConcept (Кодируемый Концепт)Тип, представляющий концепт с помощью обычного текста и/или одного или нескольких Codings
QuantitySpecial case: has system and code elements for carrying a code for the type of unit

Note: generally the choice of data type is dictated by the resource itself. When choosing a data type for an extension, see the FHIR wiki for advice about data type choice.

Coded Values, Systems, and Value Sets

Множество кодированных значений, которые разрешено использовать в элементах одного из этих 4 типов данных, называется "value set" (набор значений). Везде, где используются эти типы данных, спецификация связывает соответствующий набор значений с элементом.

Разница между кодовой системой и набором значений является существенным различием, которое легко упустить при реализации, так как это различие часто упускается при проектировании системы. К примеру, не редкость увидеть таблицу приложения, состоящую из смешанных кодов, содержащую некоторые LOINC-коды, а также некоторые дополнительные собственные коды. Довольно часто между ними нет явного различия; тот факт, что код выглядит похожим на LOINC-код, не изменяет его происхождения.

Для обмена данными, с другой стороны, явное отслеживание источника кода является и важным, и необходимым. Для того чтобы сделать это, каждой кодовой системе, определяющей коды, присваивается URL, который идентифицирует её, и все коды, которые она определяет, по сути являются парами ("Code Pair": имя с пространством имён). Так, в случае того примера со смешанным списком из предыдущего параграфа имеются две кодовые системы: LOINC (http://loinc.org) и локальная система (скажем, был назначен такой URL: http://example.com/codesystems/additional-test-codes). Таблица приложения - это единый набор значений (набор кодированных пар Code Pairs), включающий в себя коды из каждого из этих двух пространств имён. Самому набору значений назначается свой собственный URL в качестве идентификатора (например "http://example.com/fhir/ValueSet/test-codes)") - он идентифицирует набор кодированных пар (Code Pairs), но никогда не используется в качестве пространства имён в самой кодированной паре или экземпляре. В FHIR кодированные пары (Code Pairs) всегда представляются в виде пары "code" и "system", за исключением простого типа данных "code", в котором пространство имён (например элемент/свойство system) фиксировано в схеме и не представляется явно.

Choosing a system

The URL in a system is always a reference to a code system, not to a value set. The system ensures that codes can be unambiguously traced back to their original definition, and that logical comparisons, matching and inferences can be performed consistently by different systems. For this reason, choice of the correct URI for the system attribute is critical.

The correct value to use in the system for a given code system can be determined by working through the following list, in order:

  • the specification Code System Registry - if a code system is listed here, it SHALL be used
  • A system URI or OID defined as the correct value to use in FHIR by the publisher of the code system
  • the FHIR community code system registry - if a code system is listed here with status = active, it SHALL be used
  • an OID registered in the HL7 OID registry - if a code system is registered here, the OID SHOULD be used (using the syntax urn:oid:[oid])

If a code system is resolved by this list, and there is no publisher to consult, implementers will have to choose a URI to use. The priority should be to choose a unique value that won't accidently be used by another implementer for a different purpose - or a very similar purpose with a different scope.

For publishers of code systems, the following considerations should be kept in mind when defining the correct URI to use:

  • Once defined, the URI will require agreement from all implementers to change, and some may not be able to change (stored resources). If the set of users is not closed, it will usually not be possible to change the URI
  • Implementers strongly prefer a human readable URI. http://acme.com/patients/mrn is a great deal easier to work with than urn:oid:1.2.3.4.5.6.7
  • An http: address SHOULD resolve to some useful description of the code system. Ideally, if a user makes a request of the address with the media type set to a FHIR media type, the server will respond with a FHIR Value Set resource with an inline code system, but some other human or computable definition is allowed
  • http addresses should be permalinks which may re-direct to the current correct content
  • A scope of the code system URI and the correct usage of codes and displays in it's namespace SHOULD be clearly defined. See examples for SNOMED CT, RxNorm, LOINC, NDC
  • Generally, allocation of URLs is heirarchical, and most care is required in choosing the root URL. One sub-URL policies are clearly defined, URIs can often be automatically assigned

Note: if the code system is made available packaged inside a ValueSet resource, the correct URL for the system value is ValueSet.define.system, not ValueSet.uri.

Controlling the use of Coded Values

Когда у элемента есть привязка к набору значений, у элемента "binding" могут быть следующие свойства:

NameОписательное имя, используемое при показе информации о привязке
StrengthКак следует понимать эту привязку - см. ниже
ReferenceURL набора значений. Как правило, это прямая ссылка на ресурс ValueSet, но может быть и непрямой ссылкой, подразумевающей набор значений
DescriptionТекстовое описание использования кодов. Если элемент "reference" не заполнен, элемент "description" должен быть заполнен обязательно. Если элемент "reference" заполнен, то в "description" можно поместить дополнительные заметки по использованию и реализации набора значений

В декларативных типах данных FHIR привязка всегда оформляется с помощью ElementDefinition.binding.

Value Set References

В спецификации есть много мест, где используются ссылки на наборы значений для привязки кодированного значения к набору значений (value set):

ElementDefinition.binding.valueSet[x]Используется для привязки заданного элемента к набору значений
ConceptMap.source[x] and .target[x]используется для указания области действия мэппинга в Concept Map - из одного набора значений в другой
Questionnaire.group.question.optionsОбозначает, что ответы на ряд вопросов нужно брать из набора значений
ValueSet.compose.importСодержимое набора значений также включает в себя содержимое импортированного набора значений
ValueSet Reference ExtensionОбозначает, что конкретное кодированное значение было выбрано из указанного набора значений

В этом списке используются два типа ссылок на наборы значений, прямые и логические.

Прямые ссылки на наборы значений

Прямая ссылка на набор значений имеет тип Reference и ссылается непосредственно на ValueSet на основе URL, как правило, на терминологический сервер с FHIR RESTful API. При обращении к набору значений по такому типу ссылки, система должна получить доступ непосредственно к URL (после конвертации относительной ссылки в абсолютную ссылку в соответствии с локальным контекстом). Если этот процесс не удается, система не в состоянии разрешить ссылку на набор значений и должна соответствующим образом обработать ошибку.

Пример: 

GET fhir/Questionnaire/234

<Questionnaire>
  ...
  <question>
    <options>
      <reference value="ValueSet/234234"/>
    </options>
  </question>
  ....
</Questionnaire>

Здесь указывается, что значения для конкретного вопросника идут из набора значений с идентификатором 234234 в той же FHIR-точке взаимодействия. Для разрешения этой ссылки система выполнит GET fhir/ValueSet/234234

Обычно прямые ссылки как эта подходят для внутрипроцессных ссылок, в закрытых или тщательно контролируемых экосистемах. В более общем контексте эти ссылки склонны к ломкости с течением времени, так как веб-URL - включая RESTful API URLS - лекго переназначаются. По этой причине системам поощряется использовать логические ссылки на наборы значений.

Логические ссылки на наборы значений

Логическая ссылка на набор значений имеет тип uri, где указывается абсолютный URI, который соответствует тому, что указывается в ValueSet.url. URL набора значений может быть - и предпочтительно, чтобы был - веб-адресом, который на самом деле разрешается прямо в фиксированный веб-адрес, который служит в качестве официального источника для этого набора значений. Кроме того, система может запросить терминологический сервер(-ы) разрешить набор значений по этому URL в качестве его идентификатора.

Пример: 

<StructureDefinition>
  ...
  <element>
    ...
    <binding>
      ...
      <valueSetUri value="http://hl7.org/fhir/vs/clinical-findings"/>
    </binding>
    ...
  </element>
  ....
</StructureDefinition>

Здесь указывается, что элемент привязан к набору значений с Value.url = http://hl7.org/fhir/vs/clinical-findings. Один из способов получить доступ к этому набору значений - это попытаться выполнить запрос GET http://hl7.org/fhir/vs/clinical-findings, который работает, для этого набора значений - http://hl7.org/fhir/vs/clinical-findings вернет официальный набора значений для этого URL.

Кроме того, набор значений может быть разрешен с помощью локального терминологического сервера. Если это работающий Терминологический сервер FHIR, то это будет работать следующим образом: 

GET fhir/ValueSet?url=http://hl7.org/fhir/vs/clinical-findings

Если терминологическому серверу известен указанный набор значений, то он вернет этот набор значений. Если URL не разрешается в официальный набор значений, и терминологическому серверу(-ам) не известен этот набор значений, система не в состоянии разрешить такой набор значений и должна обработать ошибку соответствующим образом.

В качестве URL набора значений допускается использовать URI, к примеру, UUID (например urn:uuid:c0e0d027-1250-4278-8f44-33a49dc67916). К таким наборам значений нельзя получить прямой доступ, они должны приходить из терминологического сервера. Обратите внимание, что данная спецификация определяет много наборов значений, имеющих неразрешаемый логический URL (примеры из SNOMED CT, RxNorm, LOINC).

Использование логической ссылки, являющейся прямой ссылкой на официальный набор значений - это простейший и наиболее надежный подход. Однако он требует подходящих настроек хостинга и не всегда может быть гарантирован, поэтому он не является обязательным.

Версионные логические ссылки

Идентификатор набора значений содержит две части: URL-адрес и версию. Некоторые наборы значений всегда имеют только одну "версию"; правка содержимого такого набора значений приведет к присвоению нового URL. Другие, однако, поддерживают один и тот же URL, но меняют версию. Терминологический сервер может иметь несколько наборов значений для одного ValueSet.url с различными версиями.

Для уточнения, на какую версию набора значений идет ссылка, добавьте версию к логическому URL следующим образом: 

<valueSetUri value="http://hl7.org/fhir/vs/clinical-findings?version=0.8"/>

Это ссылка на конкретную версию набора значений. Поиск такого набора значений на терминологическом сервере будет выглядеть следующим образом: 

GET fhir/ValueSet?url=http://hl7.org/fhir/vs/clinical-findings&version=0.8

Заметим, что если ссылка на набор значений не имеет версии, и сервер находит несколько версий для набора значений, система, использующая этот набор значений, должна выбрать последнюю версию этого набора значений и использовать его.

Unbound

Note that as a matter of ongoing development, a few elements that have coded data types are not bound to any value set at all. Bindings are to be provided for these elements.

Степень привязки

Практически все элементы, которые имеют кодированный тип данных, привязываются к набору значений. Эти привязки различаются по степени гибкости: насколько жестко необходимо следовать набору значений:

requiredДля совместимости экземпляры этого элемента ДОЛЖНЫ включать код из заданного набора значений
extensibleДля совместимости экземпляры этого элемента должны включать код из указанного набора значений, если есть подходящий код в этом наборе значений для передаваемого концепта. Если набор значений не содержит подходящего для этого концепта кода (с точки зрения человека), то вместо него могут быть включены альтернативные кодинги (из различных кодовых систем, включая локальные) или (data type allowing) текст.
preferredПоощряется в экземплярах использовать указанные коды в целях совместимости, но не обязательно делать это, чтобы они считались совместимыми
exampleНе ожидается и даже не поощряется в экземплярах использовать указанный набор значений. Такой набор значений просто показывает примеры типов концептов, предназначенных для использования

Независимо от степени привязки, когда для описания локального применения используется StructureDefinition, он может привязать элемент к другому набору значений, чтобы более точно указать, какие именно кодированные значения могут быть использованы для этих элементов, и/или повысить силу привязки. Для этого существуют различные правила в зависимости от степени привязки, как описано ниже. Вообще ожидается, что юрисдикции, проекты и поставщики будут работать вместе, чтобы выбрать актуальные рабочие наборы значений.

Тип Required (обязательная привязка)

Чтобы считаться совместимыми, экземпляры этого элемента ДОЛЖНЫ включать код из указанного набора значений. .

В стандарте она, как правило, используется для элементов, где значение должно строго контролироваться, чтобы все могли интерпретировать его с уверенностью. Обычно такой тип используется для элементов с типом code:

  • элемент связан с набором значений, который содержит список различных кодов из заданной системы (и версии, где это необходимо)
  • элемент связан с каким-то внешним стандартом, который определяет набор допустимых кодов, которые могут быть использованы (типичные примеры ссылок: Mime Types, Language Codes, UCUM)

Другое место, где это используется, это при профилировании ресурсов, и когда есть соглашение для конкретного контекста использования, что конкретный набор кодов являются единственными, которые могут быть использованы. В этих случаях тип данных должен содержать одно из значений из набора значений. Если тип данных CodeableConcept, то одно из значений coding должно быть из указанного набора значений. Также может быть предоставлен Text, что всегда рекомендуется, но не является приемлемой заменой необходимого кода.

Обратите внимание на дополнительные правила относительно обязательных привязок при использовании с типом данных code:

  • Там, где набор значений определен стандартом FHIR, список допустимых кодов будет зафиксирован в XML-схеме
  • Сравнение между кодами всегда чувствительно к регистру, если только эти коды не берутся из другой спецификации, где явно указано иное
  • Список разрешенных кодов может быть расширен только в последующих версиях спецификации FHIR

Когда элемент привязан к обязательному набору значений, производные профили могут устанавливать правила использования этих кодов, но не могут определять новые или дополнительные коды для этих элементов.

Тип Extensible (расширяемая привязка)

Чтобы считаться совместимыми, экземпляры этого элемента ДОЛЖНЫ включать код из указанного набора значений, если хоть один из кодов в наборе значений подходит к передаваемому концепту. Если набор значений не покрывает этот концепт (с точки зрения человека), вместо него можно использовать альтернативный system.code.

If the data type is CodeableConcept, then one of the coding values SHALL be from the specified value set if a code applies, but if no suitable code exists in the value set, alternate code(s) may be provided in its place. If no codes, including local codes, are available, then just text may be used.

If the data type is Coding, then the code/system SHALL be from the specified value set if a code applies, but if no suitable code exists in the value set, an alternate code may provided in its place.

Identified gaps in value sets should be submitted to the organization administering the value set in order to improve interoperability in the future.

Расширяемые привязки используются, когда имеется соглашение на уровне спецификации или профилировании о кодированных значениях, которые следует использовать, но невозможно создать ограниченный список кодов, который покроет все возможные случаи использования, в том числе те, что ещё возникают.

Когда элемент привязан к расширяемому набору значений, производные профили могут устанавливать правила использования этих кодов, но не могут определять новые или дополнительные коды для этих элементов, если только в базовом наборе значений не отсутствуют коды с подходящим значением.

Привязки типа Preferred (предпочтительная привязка)

Поощряется брать значения из указанных кодов в целях совместимости, но это не обязательное требование совместимости.

Если тип данных CodeableConcept, тогда одно из значений coding ДОЛЖНО быть из указанного набора значений, однако другой код и/или текст также может быть использован взамен.

Предпочтительные привязки используются, когда есть консенсус на уровне спецификации о кодированных значениях, которые лучше использовать, но есть осознание того, что в некоторых контекстах реализации рекомендуемые коды не могут быть использованы по различным причинам. Приложения должны рассматривать вопрос о применении предпочтительного набора значений везде, где это возможно, так как эти предпочтительные наборы значений скорее всего послужат целям совместимости в будущем.

Когда элемент привязан к предпочтительному набору значений, производные профили могут привязывать этот элемент к любому набору значений на свой выбор.

Привязки-примеры

От экземпляров не ожидается и даже не поощряется брать значения из указанного набора значений. Такой набор значений приводится всего лишь в качестве примера типов концептов, которые могут в него входить.

Примерные привязки используются, когда у элемента слишком широкое значение (например List.code), или нет соглашения о том, какие коды правильно использовать. Для таких привязок:

  • Coding: значения система/код МОГУТ быть кодами из набора значений.
  • CodeableConcept: один из кодируемых элементов МОЖЕТ содержать систему/кодиз набора значений.

Некоторое другое кодированное значение МОЖЕТ быть использовано или (для CodeableConcept) МОЖЕТ быть предусмотрен альтернативный текст. Наборы значений-примеры представлены для того, чтобы помочь разработчикам понять правильное использование элемента. Наборы значений на основе таких кодовых систем как SNOMED CT, которые имеют строгие условия лицензионного соглашения, будут приводиться в базовой спецификации FHIR только в привязках-примерах, хотя руководства по реализации для конкретной юрисдикции могут применять наборы значений, требующие лицензий.

Когда элемент привязан к набору значений-примеру, производные профили могут привязывать этот элемент к любому набору значений на свой выбор.

Другие примечания

  • Последующие версии FHIR могут заменять наборы значений-примеры на предпочтительные привязки при достижении единодушного мнения
  • Привязки, приведенные в спецификации, всегда относятся к тому набору значений, который опубликован в той же версии спецификации. Набор значений может быть ограничен списком перечисленных кодов, или может включать коды по их свойствам вместе с не зависящей от версии ссылкой на их кодовую систему, в этом случае список допустимых концептов может измениться со временем.

Binding String Values

In a few special cases, humans customarily use codes directly for elements that have type "string". A typical case is codes for states, and there are several places where a URIs must come from a set of controlled values. An element of type string or uri can also be bound to a value set. When a string or URI is bound to a value set, the value property SHALL contain the code specified by the value set, and the system and display values are ignored.
