Назначение

FHIR-реализация USLab Report сосредоточена на определении требований, спецификаций и стандартов, а также на предоставлении указаний по реализации отчетов лабораторных результатов исследований для провайдеров амбулаторной помощи в США с помощью the FHIR in the RESTful framework. Ее область действия включает требования по обеспечению встраивания результатов клинико-лабораторных исследований в систему ЭМК в виде стандартизованных структурированных данных, используя определенную межорганизационную лабораторную транзакцию. Требования такого сценария использования направлены на сообщение результатов лабораторных исследований между лабораторной информационной системой (ЛИС) и амбулаторной системой ЭМК в различных организационных объектах, например различных корпоративной структуре, собственности или управлении. Будущие версии этой реализации могут быть согласованы с существующими руководствами для расширения совместимости лабораторных результатов в разных условиях оказания помощи, например неотложная медицинская помощь и общественное здравоохранение.

Целевая аудитория

Данное руководство создано для использования аналитиками и разработчиками, которым требуется руководство по FHIR-ресурсам, элементам и особым расширениям, связанным с инициативой по интерфейсу лабораторных результатов (LOI) и HL7 V3 Lab Normative Standard. Пользователи данного руководства должны быть знакомы с деталями FHIR-спецификации и обработки ресурсов. Это руководство не предназначено для освещения этой темы.

Условные обозначения

Это руководство придерживается следующих конвенций:

  • Руководство построено с предположением, что реализатор имеет доступ к FHIR-спецификации. Хотя некоторая информация из стандарта и включена в это руководство по реализации, большинство информации из стандарта здесь не повторяется.
  • Руководство построено с предположением, что разработчик БУДУЕТ реализовывать FHIR RESTful API
  • Правила, определенные в FHIR Conformance Profile, были использованы для документации вариантов использования и ограничений, применяемых к ресурсам, описанным в этом руководстве.

Ключевые технические решения

Архитектура профилей и компонентов

Данное руководство использует профили FHIR для определения минимального набора требований в целях обеспечения возможности успешного обмена лабораторными заказами для следующих ресурсов:

  • DiagnosticOrder
  • DiagnosticReport
  • Organization
  • Media
  • Organization
  • Patient
  • Practitioner
  • Specimen

Основная задача - обеспечить обмен лабораторными заказами между ЭМК и лабораторными системами с минимальными модификациями, если они вообще будут иметь место, от одной комбинации программного обеспечения к другой, при этом поддерживая гибкость для обеспечения возможности предоставить больше возможностей разработчикам ПО, использующим одни и те же основные определения ресурсов.

Использование стандартных глоссариев

Данное руководство призывает к использованию определенных стандартных глоссариев для обмена лабораторной информацией, таких как LOINC и SNOMED. Стандартные глоссарии, в частности кодированные лабораторные тесты и их результаты, обеспечивают автоматическую поддержку принятия решений в здравоохранении пациентов, а также в надзоре за здравоохранением населения. Ресурсы Value Set предоставлены в рамках этой реализации.

Режим снимка состояния (Snapshot)

FHIR функционирует только при использовании режима снимка состояния - обновления передаются отправкой полной копии экземпляра, заполненного новыми данными. Если необходима коррекция и/или обновление статуса ресурса Observation, DiagnosticReport со всеми ссылками на все ресурсы Observation, даже если он был уже ранее отправлен, должен быть отправлен повторно с коррекцией и/или обновлением статуса и/или текущих значений. Например, когда заказывается клинический анализ крови с ручным подсчётом форменных элементов (Complete Blood Count with manual differential), анализ крови будет выполнен, а затем некоторое время спустя будет выполнен и опубликован ручной подсчёт форменных элементов. Когда анализ крови опубликован, DiagnosticReport будет ссылаться только на ресурсы Observation, как на окончательные результаты. Когда ручной подсчёт форменных элементов будет завершен, DiagnosticReport будет ссылаться на все предыдущие ресурсы Observation, а также на новые ресурсы Observation - в данном случае на анализ крови и дифференциальные результаты.

Область действия реализации

Из-за большого разнообразия систем-получателей и их потребностей данное руководство специально не указывает для каждого поля, сохранять его или нет. Это остается на усмотрение каждой отдельной системы в зависимости от ее области действия и назначения.

Сценарий использования - Амбулаторная помощь

Этот вариант использования был разработан совместными усилиями HHS/ONC стандартов и инициативой фреймворка лабораторных заказов, Фондом здравоохранения Калифорния и рабочей группой HL7 Orders and Observations.

Область действия

Область действия - отправка лабораторных результатов из лаборатории провайдеру амбулаторной помощи.

В области действия

  • Определение основных элементов данных, требуемых для амбулаторных результатов клинико-лабораторных исследований.
  • Сообщение результатов клинико-лабораторных исследований при оказании амбулаторной помощи в США.
  • Отправка результатов клинико-лабораторных исследований в виде стандартизованных структурированных данных, чтобы их можно было встроить таким образом в систему ЭМК.
  • Сообщение результатов исследований по заказу, который был размещен вручную или электронным образом.
  • Некоторые специфичные для заказа данные были включены для обеспечения возможности ЭМК-системам-получателям соотносить результаты с первоначальными заказами.
  • Покрытие всех требований CLIA к составлению отчетов.
  • Получение лабораторных результатов лицом, не являющимся непосредственным заказчиком.

Вне области действия

  • Спецификации и руководство по реализации транзакций лабораторных заказов (см. USLabOrder)
  • Запрос результатов лабораторных исследований.
  • Запрос истории лабораторных результатов.
  • Прием исторических результатов лабораторных исследований.
  • Вторичное использование лабораторных данных (например использование сообщенных результатов лабораторных исследований для контроля общественного здравоохранения или биологических наблюдений).
  • Заказ и сообщение лабораторных результатов в госпитале.

Действующие субъекты

Есть два действующих субъекта, имеющие обязанности, связанные с профилями соответствия, определенными в данном документе:

  • Отправитель лабораторного отчета - создатель экземпляра ресурса DiagnosticReport, который декларирует соответствие "RESTful FHIR" и профилям FHIR, описанным в данном руководстве. Этим действующим субъектом является ресурс Organization, на который в FHIR ссылаются через элемент .performer (Organization).
  • Получатель лабораторного отчета - получатель экземпляра ресурса DiagnosticReport, который декларирует поддержку "RESTful FHIR" и FHIR-профилей, определенных в этому руководстве. Этот действующий субъект подразумевается в FHIR (т. е. не упоминается явным образом в экземпляре ресурса DiagnosticReport или связанных ресурсах).

Пользовательская история

Провайдер (заказчик) может ввести лабораторный заказ в амбулаторную систему ЭМК. Генерируется лабораторная заявка (бумажная или электронная) и передается в лабораторию. Информация в лабораторной заявке вводится вручную или фиксируется в электронном виде в ЛИС. После того, как образец(-цы) для анализа был взят и, если необходимо, выслан или доставлен в лабораторию, лаборатория обрабатывает этот образец(-цы). Если образец подходит для проведения исследования, лаборатория выполнит исследование. До момента успешного завершения исследования может потребоваться дополнительная коммуникация для индикации отмены, неспособности выполнить исследование и связанных с этим причин; например когда образец либо не подходит для выполнения заказанного теста, либо является неудовлетворительным в другом отношении, об отклонении образца будет передано с помощью профиля USLabOrder. Лаборатория выполняет или пытается выполнить тест(-ы). Если тестирование проходит успешно, результаты получены и введены/опубликованы в ЛИС. Уполномоченное лицо в лаборатории проверяет и утверждает результаты лабораторного исследования, либо рецензент заверяющей лаборатории проверяет запись в случае автоматического процесса верификации, которые будут отправлены провайдеру и всем не-провайдерам заказа, указанным в лабораторном заказе. Результаты либо pushed or pulled в систему ЭМК провайдера (получателя результатов). Эта система ЭМК включает результаты в электронную карту пациента. Провайдер логинится в свою систему ЭМК и просматривает лабораторные результаты для обоснования решений по уходу за пациентом.

Допущения, сделанные в сценарии использования

  • Провайдеры безопасным способом получают доступ к клинической информации через EHR-систему.
  • Подходящие протоколы - транспортный и безопасности, методика идентификации пациентов, методика идентификации заявки (заказа); согласие пациента; процедуры обеспечения конфиденциальности и безопасности; кодирование, глоссарий, обработка ошибок и стандарты нормализации были согласованы всеми участниками.
  • Этот сценарий использования . только обмена лабораторными результатами, которые соотносятся с лабораторными исследованиями, входящими в область действия.
  • Данный сценарий использования покрывает все требования CLIA по передаче отчетов
  • Для процесса взятия образцов для анализа предполагается, что данные, включенные в dataset considerations таблицу 6, будут доступны и представлены в самом результате.
  • Юридические вопросы и проблемы управления, касающиеся разрешений на доступ к данным, прав собственности на данные и использования данных, находятся в силе.
  • Установленная инфраструктура сети и политики безопасности существует для того, чтобы сделать возможным непротиворечивый, надлежащий и точный обмен данными между системами-провайдерами, репозиториями данных и сервисами поиска. Это включает в себя, но не ограничивается:
    • Методы идентификации и аутентификации пользователей;
    • Методы идентификации и определения поставщиков медицинских услуг;
    • Методы обеспечения соблюдения политики авторизации доступа к данным;
    • Методы обеспечения достоверности данных;
  • Подробные контрольные записи хранятся по мере необходимости всеми системами-участниками.
  • Политика безопасности и конфиденциальности, процедуры и практики, как правило, реализуются для поддержки приемлемого уровня безопасности и конфиденциальности пациентов; т. е. HIPAA, HITECH и критерии сертификации EHR (ЭМК).
  • ЛИС будет источником результатов лабораторных исследований, а ЭМК будет получателем.
  • Данный сценарий использования осознает различия в требованиях к передаче отчетов между местными, государственными, родовыми и территориальными границами, также как и в добровольности или обязательности требований.
  • Лаборатории отвечают критериям аккредитации в соответствии с требованиями юрисдикции или критериями ведомств.

Предусловия

  • Заказывающим провайдером был создан заказ на получение одного или нескольких результатов лабораторных тестов.
  • Когда это указано, лаборатория получает запрос на отправку лабораторных результатов лицу, не являющемуся непосредственным заказчиком.
  • Лаборатория получает заказ (в электронном, бумажном виде и т. п.), или лаборатория получает запрос повторно провести (повторить) тест, или определяет необходимость повторного проведения теста для возможной коррекции, или определяет, что требуется рефлексное тестирование (которое основано на критериях, установленных медицинским комитетом по надзору), или определяет необходимость внесения поправки в результат теста, содержащий ошибочную информацию.
  • Лаборатория получает надлежащую клиническую информацию для выполнения заказанного исследования.
  • Лаборатория ввела вручную или через интерфейс подходящие (или исправленные) данные из заказа в ЛИС.
  • Лаборатория получила и обработала надлежащим образом идентифицированный образец(-цы) для анализа, относящийся к заказанному тесту(-ам).
  • Лаборатория ввела или получила от заказывающей системы ЭМК уместные данные об образце в ЛИС.
  • Лаборатория выполнила заказанные тесты на полученных образцах и/или включила подсчеты и справочные/эталонные данные в полученные результаты, которыми в дальнейшем будет обмениваться.
  • Лабораторный ресурс DiargnosticReport ссылается на соответствующие ресурсыPatient и DiagnosticOrder для сопоставления лабораторных результатов с нужным пациентом и исходным заказом.
  • ЛИС способна и готова отправить лабораторные результаты, используя FHIR, REST и стандартизованный структурированный формат данных.
  • ЭМК-система готова к использованию и способна принимать лабораторные результаты, используя FHIR, REST и стандартизованный структурированный формат данных.
  • Лабораторный результат проверен и готов к выпуску в свет.

Постусловия

  • Лабораторные результаты точно запротоколированы и успешно переданы в электронном виде из ЛИС в ЭМК-систему, модуль или другой приемник результатов провайдера заказа (разместившего заказ).
  • ЭМК-система провайдера получила электронным образом лабораторные результаты, включила их в стандартном структурированном формате и, если возможно, связала с пациентом и лабораторным заказом.

Обработка ошибок

Обратитесь к спецификации FHIR за REST статус-кодами и использованием ресурса OperationOutcome, если потребуется дополнительная информация о транзакционных ошибках. Примечание для голосования: спецификация обработки ошибок в настоящее время не в полной мере определена для этой реализации.

Кодовые системы и наборы значений

Для успешной реализации обмена сообщениями требуется, чтобы передаваемые сообщения (экземпляры сообщений) содержали в кодированных полях допустимые значения. Для дальнейшего обсуждения использования кодов в FHIR обратитесь к спецификации FHIR.

LOINC

Использование кодовой системы Logical Observation Identifiers Names and Codes (LOINC) требуется там, где для поля Observation.name есть соответствующий доступный LOINC-код, т. е. являющийся результатом. Подходящий статус определен в разделе 11.2 Classification of LOINC Term Status руководства по LOINC. Если используется локальная кодовая система, локальный код также должен быть послан в дополнение к LOINC, чтобы помочь с определением проблем кодирования. Когда подходящего LOINC-кода не существует, можно отправить только локальный код.

Хотя требования к хранению данных в ЭМК-системах не рассматриваются в этом руководстве, рекомендуется, чтобы LOINC-коды хранились в ЭМК или были ей доступны, по следующим причинам:

  1. Если результат относится к отчетному состоянию, и лаборатория предоставляет LOINC-код,
  2. Meaningful Use Stage требует от ЭМК отправлять LOINC-код в общественное здравоохранение. Если в лабораторию отправляется только LOINC-код в OBX-3, тогда ЭМК должна сохранить и поддерживать этот код для удовлетворения требований CLIA к отчетности.
  3. Коды LOINC могут использоваться в целях вторичного обмена данными и других партнерских соглашений по обмену.

SNOMED CT

Использование кодовой системы SNOMED-CT требуется там, где для поля Observation.valueCodeableConcept есть доступный SNOMED-CT-код концепта, т. е. непосредственно кодируемый результат. Если используется локальная кодовая система, локальный код также должен быть послан в дополнение к SNOMED-CT-коду концепта, чтобы помочь с определением проблем кодирования. Когда подходящего SNOMED-CT-кода не существует, можно отправить только локальный код. Для удовлетворения требований CLIA к отчетности также рекомендуется использовать элемент Observation.valueCodeableConcept.text, если исходный текст от лаборатории, который используется для печати и/или отображения, отличается от значения для отображения кода.

SNOMED-CT требуется в элементах для указания терминов источника образца для анализа Specimen.type и Specimen.Collection.sourceSite там, где доступен SNOMED-CT-код концепта.

UCUM

Использование кодовой системы UCUM (Unified Code for Units of Measure) требуется для сообщения единиц измерений в поле Observation.quantity.

Наборы значений (Value Sets)

Наборы значений для этой реализации, как для USLabOrder, так и USLabPHReport, находятся на странице Реализация USlab. Кроме того, также доступны мэппинги на HL7 Version 2.5.1 Implementation Guide: S&I Framework Lab Results Interface.

Подтверждающее и рефлексное тестирование (включая тестирование на чувствительность микроорганизмов)

Определение: Дополнительное лабораторное тестирование, входящее в первоначальный запрос на исследование со ссылкой на конкретное дополнительное тестирование, например "Urinalysis w/Culture Reflex" в отличие от "Urinalysis", заказанному как отдельный тест. Решение о выполнении рефлексного или повторного тестирования принимается на основании результатов первоначального тестирования и применения установленного локального или государственного практического руководства, утвержденного протокола или юридического требования.

  • Пример: Анализ мочи с повышенными лейкоцитами сигнализирует о потенциальной бактериальной инфекции, и заказывается подтверждающий анализ мочи на культивацию на том же образце для анализа в качестве рефлексного тестирования. В зависимости от стандартной рабочей процедуры лаборатории, ЛИС и характера рефлексного или подтверждающего тестирования может быть создано одно или несколько следующих: новый инвентарный номер, новые коды исследований и дополнительные расходы.
  • CLIA Compliance: Исходный запрос на тестирование, полученный лабораторией, подходит для демонстрации заказа как для начального, так и дополнительного тестирования в соответствии с CLIA и целями аудита CMS.
  • Процесс ЛИС: ЛИС передает отчет по рефлексному тестированию в одном из следующих видов:
    1. В виде дополнительных результатов в том же DiagnosticReport, ссылающемся в элементе DiagnosticReport.requestDetail на тот же самый DiagnosticOrder из оригинального заказа
    2. В виде одного или нескольких дополнительных DiagnosticReports, ссылающихся в элементе DiagnosticReport.requestDetail на тот же самый DiagnosticOrder из оригинального заказа
    3. В виде одного или нескольких дополнительных DiagnosticReports, ссылающихся в элементе DiagnosticReport.requestDetail на новый DiagnosticOrder из добавленного заказа и на DiagnosticOrder из оригинального заказа

Дополнительное тестирование

Определение: Дополнительное лабораторное тестирование заказывается уполномоченным провайдером (в соответствии с CLIA и государственным законом) над существующим образцом для анализа после того, как первоначальный заказ на исследование был отправлен в лабораторию. Решение о запросе на проведение дополнительного исследования принимается провайдером индивидуально и на основе любого числа факторов, не ограниченных результатом теста.

  • Пример: Врач заказывает клинический анализ крови и основную метаболическую панель для амбулаторного больного, который находится в кабинете с симптомами усталости и субфебрильной температурой после туристического похода в Висконсин. После консультации со специалистом по инфекционным заболеваниям позже этим же днем он звонит в лабораторию и заказывает дополнительный тест на антитела к болезни Лайма для образца для анализа, уже находящегося в лаборатории.
  • Соответствие CLIA: CLIA требует от лаборатории получать письменный или электронный запрос на дополнительное исследование от уполномоченного провайдера для своих записей. Если запрос на тестирование был устным, лаборатория обязана задокументировать свои попытки получить письменный или электронный запрос на исследование в течение 30 дней. [§42CFR493.1241(b)]

Соображения по резолюции к улучшению клинической лаборатории

В США клинико-лабораторные исследования образцов для анализа, взятых у людей, регулируются резолюцией к улучшению клинической лаборатории (Clinical Laboratory Improvements Amendments (CLIA)) 1988 года. Несколько разделов постановлений, реализующих CLIA, влияют на то, как электронные лабораторные данные форматируются для США, и они описаны в данном разделе. Затронутые области включают обязательные требования к заказу тестов. CLIA Regulation Specifics.

Обязательные требования для заказа

Раздел 493.1241 положения CLIA обязывает лабораторию иметь письменный или электронный запрос на исследование пациента от уполномоченного лица, а также определяет пункты, которые должны присутствовать в рамках запроса на клинико-лабораторное исследование. Лаборатория может принимать устные запросы на лабораторные тестирования, если она требует письменную или электронную авторизацию в течение 30 дней с момента устного обращения и поддерживает авторизацию или документацию попыток получения этой авторизации.

Пояснительные указания к элементам, обязательным в заявке на исследование. Поля, на которые оказывается влияние, включают следующие:

Patient.identifier Требуется уникальный идентификационный номер пациента
Patient.name Имя или уникальный идентификатор пациента.
Observation.name Обязательна однозначная идентификация выполненного теста. Настоятельно рекомендуется использование LOINC-кодов для дополнительных тестов. Для некоторых тестов CLIA требует дополнительную информацию: лаборатории, использующие инструменты производителя, комплекты или тест-системы, маркированные "только для исследований" или "только для исследования", должны четко указывать, что результаты исследований не должны быть использованы для лечения или в целях диагностики. Observation.interpretation Должны быть указаны единицы измерения, если необходимо, или интерпретация. Для таких тестов, как генетический скрининг, интерпретация может быть фактическим результатом теста.
Observation.referenceRange Когда имеется, необходимо указывать диапазон нормальных значений.

CLSI-определения - количественные, полуколичественные, качественные результаты

Следующие определения были получены с веб-сайта CLSI:

  • КОЛИЧЕСТВЕННЫЙ
    • Характеристика, используемая для лабораторных тестов, которые дают результаты, выраженные в числовом количестве или уровне (концентрации) аналита в образце;
      ПРИМЕЧАНИЕ 1: Обычно сравнивается с аккредитованным признанным стандартом;
      ПРИМЕЧАНИЕ 2: В противоположность качественным тестам.
    • При использовании для описания теста означает тест, который дает числовой результат. Например тест на содержание глюкозы в крови у постели больного может сгенерировать результат 120 mg/dL (1.20 g/L). В противоположность этому, качественный тест дает не-числовой результат, такой как "положительно" или "обнаружены". Подмножество количественных тестов, называемых полуколичественными, дает результаты либо в диапазоне значений, например urine dipstick, результатом которого могут быть диапазоны глюкозы 0-40, 40-100 и >100 mg/dL (0-0.4, 0.4-1 и >1 g/L), или в виде серии относительных значений, например тот же urine dipstick, результатом которого может быть показатель гемоглобина 0, +, ++, +++ и ++++.
  • КАЧЕСТВЕННЫЙ
    • При использовании для описания теста означает, что тест дает описательный результат, а не числовой. Например тест мочи на беременность может генерировать «позитивный» или «негативный» результат для мочевого ХГЧ. В противоположность этому, количественный тест генерирует числовой результат. Процедуры контроля качества и отчетности существенно отличаются для количественных и качественных тестов.
    • Характеристика, применяемая к лабораторным тестам, которые обнаруживают и/или идентифицируют определенный аналит, компонент или состояние:
      ПРИМЕЧАНИЕ 1: Этот термин применим к тестам, которые обнаруживают, присутствует или отсутствует определенный аналит, компонент или состояние, и ему иногда присваивается положительная степень (т. е. 1+, 2+);
      ПРИМЕЧАНИЕ 2: Иногда называются полуколичественными тестами;
      ПРИМЕЧАНИЕ 3: Может осуществляться конкретное определение.
  • ПОЛУКОЛИЧЕСТВЕННЫЙ
    • Тест, имеющий градиент (зависимость) "доза-ответная реакция", который может включаться в отчетный результат, но для которого не существует официальных градуировочных шкал для определения погрешности и неточности; тесты, которые дают результаты в приблизительном диапазоне значений (например отслеживается, умеренный);
      ПРИМЕЧАНИЕ: Это определение включает в себя тесты с субъективным отсчетом количественного анализа, такие как IF-ANA титры, а также включает в себя тесты с инструментальным отсчетом количественного анализа, такие как ELISA-ANA, когда шкала измерительного прибора не может быть отнесена к официальным градуировочным шкалам.
    • Тесты, которые дают результаты в приблизительном диапазоне значений (например отслеживается, умеренный).

Глоссарий

  • Analyte (Вещество, определяемое при анализе): Компонент, представляемый в названии измеряемой величины. Это наиболее гранулированный уровень, на котором выполняются измерения, он всегда представляется с помощью отдельной группы сегментов Observation.
  • Cancellation (Отмена): Акт отмены заказа.
  • Electronic Health Record (Электронная медицинская карта): Клиническая информация о конкретном пациенте, которая хранится в электронном виде в системе ЭМК. Система электронных медицинских карт. Данное руководство использует этот термин в том же контексте, как указано в "HL7 EHR System Functional Model White Paper" разделе 4 Определения (HL7 2004 www.hl7.org): "Важно отметить, что DSTU не пытается установить другое определение EHR-систем, однако выбирает для применения существующие определения, которые включает концепцию ЭМК-систем как систему (по крайней мере одну) или систему систем, которые совместно удовлетворяют потребностям конечного пользователя."
  • Future Order (Будущий заказ/Заказ на срок): Будущий заказ - это заказ с датой/временем начала, где эта дата/время начала обозначает самое раннее время, когда можно будет взять образец для анализа.
  • Laboratory (Лаборатория): Объект или организация, которая проводит лабораторные исследования образцов для анализа с целью предоставления информации для диагностики, профилактики, лечения заболевания или нарушения, или оценки здоровья для людей. Laboratory Information System (LIS) - Лабораторная информационная система (ЛИС) Информационная система, которая получает, обрабатывает и хранит информацию, относящуюся к лабораторным процессам. ЛИС может взаимодействовать с HIS и EHR приложениями. Данное определение является минимальным и опускает множество функций и возможностей, которые, как правило, связаны с лабораторными информационными системами. Эта минимальная характеристика является намеренной, чтобы включить как можно более широкий набор лабораторных информационных систем в этот сценарий использования. Минималистский характер этого определения вовсе не исключает ЛИС со значительно большими возможностями.
  • Laboratory Order (Лабораторный заказ): элемент item ресурса DiagnosticOrder, запрашивающий одно или несколько измерений (ресурсы Observation). Является синонимом Requisition (Заявка) при отсылке к отдельному пункту ресурса DiagnosticOrder.
  • Laboratory Order System (Система лабораторных заказов): Программное обеспечение, либо автономное, либо как часть EHR системы, используемое провайдером (Order Placer) для управления лабораторным заказом, в том числе создания лабораторной заявки и отправки ее в лабораторию.                 Обычно система лабораторных заказов является неотъемлемой частью системы управления заказами, что позволяет пользователям управлять заказами для различных видов услуг, процедур, расходных материалов и т. д. Так как данное руководство фокусируется только на обмене данными, связанными с лабораторными заказами, в нем намеренно используется очень ограниченное определение.
  • Laboratory Requisition (Лабораторная заявка): Набор сведений, составляющих официальный запрос на одно или несколько лабораторных исследований, которые должны быть выполнены для индивидуального пациента. Лабораторная заявка конкретизируется в клинических условиях и передается в лабораторию в виде отдельной бумаги или электронного артефакта. Лабораторные заявки всегда включают в себя, по меньшей мере, один заказ исследования. В терминах FHIR транзакции заказа, это представляет собой один или несколько элементов item ресурса DiagnosticOrder.
  • Newborn (Новорожденный): Точная граница, когда пациент считается новорожденным или младенцем, может меняться в зависимости от интерпретации, и данное руководство не предлагает окончательного ответа на этот вопрос. Для получения дополнительных сведений читатель может обратиться к следующим ресурсам:
    • Mosby's Medical Dictionary, 8th edition. © 2009, Elsevier
    • Всемирная организация здравоохранения - занимается стандартизацией перинатальных определений
    • Лечение гипербилирубинемии у новорожденных, детей или 35 и более недель беременности
    • http:/pediatrics.aappublications.org/content/114/1/297.full.html
  • Orderable Test (Заказываемый тест): Запрос на выполнение отдельного теста или панели. Это всегда относится к одному элементу item ресурса DiagnosticOrder и может иметь один или более связанных аналитов (наблюдений).
  • Panel (Панель): Хотя имеются различия в значении термина "панель" среди различных лабораторий, для целей данного руководства она определяется как группировка процедур по измерению нескольких аналитов из одного образца для анализа (или нескольких образцов в некоторых случаях) и может быть заказана с помощью одного лабораторного заказа. Также это может упоминаться как "battery". Например CBC [complete blood count - клинический анализ крови] или анализ мочи можно назвать панелями.
  • Order Set (Предустановленный набор заказов): Набор лабораторных заказов, которые включают несколько тестов и панелей и для которых может потребоваться несколько образцов, но для удобства могут быть заказаны одним блоком. Например "diabetic order set profile" (профиль набора заказов при диабете) может включать в себя клинический анализ крови, определение содержания гликозилированного гемоглобина и анализ мочи. Термин "панель" часто используется как взаимозаменяемый с "набором заказов", таким образом профиль набора заказов, который содержит несколько заказов лабораторных тестов, которые могут быть сами по себе или комбинироваться с другими заказами исследований (например радиологическое изображение, консультация и т. д.) можно считать набором заказов. Наборы заказов не должны передаваться лаборатории. Запрос на аннулирование (RFC) Запрос провайдера (Order Placer) не выполнять заказ.
  • Test (Тест/исследование): Медицинская процедура или именованный набор связанных процедур, которые включает в себя анализ одного аналита с помощью одного образца крови, мочи или других образцов, взятых у пациента, с целью диагностики заболевания или состояния здоровья, планирования или оценки лечения, или контроля за ходом болезни.
